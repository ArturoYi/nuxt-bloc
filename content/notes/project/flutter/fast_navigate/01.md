# 设计目标与项目初衷

> **阶段定位**：动手写第一行代码之前，先把「为什么要做」和「做到什么程度算赢」写清楚。  
> **代码仓库**：`packages/fast_navigator`  
> **业务接入参考**：`apps/BarrelLock/BarrelLock_android/lib/router/router.dart`

---

## 一、触发背景

Flutter Bazaar 是 Melos 8 多平台 Monorepo（Android / iOS / macOS / Web / Windows / Linux）。路由是**横切所有 App 的基础设施**，一旦选型失误，六个平台会一起还债。

当时面临的选择：

| 方案 | 优点 | 隐患 |
| --- | --- | --- |
| Navigator 1.0 | 上手快 | 栈黑盒、Web 割裂、深链难 |
| go_router | 官方维护、功能全 | 扩展点固定、特殊栈需求难深度定制 |
| auto_route | 代码生成强 | 生成物难 debug、与「零第三方路由依赖」目标冲突 |
| **自研 fast_navigator** | 完全掌控 State / Middleware / Page Key | 需真正理解 Navigator 2.0 |

**架构师判断**：Bazaar 长期维护、可能有混合栈 / 定制拦截 / 全端一致行为的需求，值得在 Navigator 2.0 四件套之上自建一层——对标 go_router 的**设计思想**，而非 fork 其代码。

---

## 二、一句话本质

> **Navigator 2.0 的本质：用不可变的「路由状态（Page List）」替代 1.0 的「命令式栈黑盒」。RouterDelegate 做唯一栈管理者，Parser 做状态 ⇄ URL 翻译器，Page 做最小渲染单元——框架只做「状态机 + 匹配 + 拦截」，不做 UI 黑盒。**

fast_navigator 在这一层之上再加：

- **NavigationState**（SSOT 唯一真相源）
- **RouteRegistry**（匹配引擎）
- **FastNavigator**（业务门面 API）
- **RouteMiddleware**（拦截管道，M3）

---

## 三、核心需求汇总

从 BarrelLock 等业务 App 倒推，整理为七条硬需求：

1. **原生 Navigator 2.0 手写**，具名路由 + 页面栈完全自主管控，禁止 `Navigator.push` 黑盒改栈。
2. **单一 NavigationState 驱动全平台**，Mobile / Web / Desktop 同一套状态机。
3. **URL 与栈双向同步**，Web 刷新、浏览器后退、深链打开有明确语义。
4. **统一返回协议**：物理返回键、手势、AppBar 返回、`FastNavigator.pop()` 走同一 `pop()` 入口。
5. **Page Key 策略规范化**，防止 Navigator diff 错乱导致状态丢失或页面不刷新。
6. **Middleware 管道**：登录守卫、权限、埋点不散落在 Widget 里。
7. **工程化**：零第三方路由依赖、可单测、可接入 Melos CI。

---

## 四、痛点 → 解法对照

| 痛点 | fast_navigator 解法 |
| --- | --- |
| 1.0 栈不可控、Web 割裂 | 单一 `NavigationState` 驱动全平台 |
| URL 与栈不同步 | `FastRouteInformationParser` ⇄ `FastRouterDelegate` 双向绑定 |
| 手势 / 物理返回乱 pop | `PopNavigatorRouterDelegateMixin` + 统一 `popRoute()` |
| Page 复用错乱 | `RouteMatch.key` 生成策略（extra 不进 Key） |
| 直接改 List 不刷新 | 强制不可变 `NavigationState`，每次栈变更返回新实例 |
| 业务 API 缺失 | `FastNavigator` 门面：`push / pop / go / replace / popUntil` |
| 登录 / 权限 / 埋点散落 | `RouteMiddleware` 管道（M3） |
| 非法路径崩溃 | `unknownRoute` + 404 Page，匹配失败不 throw |
| 连点重复跳转 | `navigationThrottleMs` 节流（见 05 篇） |
| 动画与平台不一致 | `PageTransition` + `PageFactory`（见 06 篇） |

## 五、与 go_router 的定位差异

| 维度 | go_router | fast_navigator |
| --- | --- | --- |
| 依赖 | 官方维护、功能全 | 自研、轻量、可深度定制 |
| 学习成本 | 低（封装完） | 需理解 2.0 原理（刻意为之） |
| 混合栈 / 定制拦截 | 扩展点固定 | 完全掌控 Middleware / State |
| Shell / StatefulShell | 内置成熟 | M4 路线图 |
| 适用场景 | 快速交付 | 长期维护、特殊栈需求 |

**不是替代 go_router，而是「在团队内可控的前提下，复刻其分层思想」。**

## 六、迭代路线图（当时写下的心智模型）

| 阶段 | 交付 | 验证标准 |
| --- | --- | --- |
| **M1 内核** | State + Delegate + Parser + Page + push/pop/go | Web 刷新、浏览器后退正常 |
| **M2 业务 API + 动画** | replace / popUntil + LaunchMode + PageFactory | 单元测试覆盖栈变换与过渡策略 |
| **M3 中间件** | 全局 / 路由拦截 + redirectLimit | 未登录 redirect 可测 |
| **M4 Shell** | 底部 Tab + 各 Branch 独立栈 | 对标 go_router StatefulShell |
| **M5 工程化** | 路由表拆分、日志、调试面板 | 接入 Bazaar 全端 App |

> 写路线图时的自我要求：**每一阶段都必须能跑起来一个最小 Demo**，而不是先把所有类文件建空壳。

## 七、架构师视角：三个关键决策

### 决策 1 — State 先行，UI 后行

先实现 `NavigationState` 的纯函数式栈操作（push/pop/go），再碰 `RouterDelegate`。  
理由：栈变换逻辑可单测，不依赖 Widget 树——这是 2.0 与 1.0 最大的工程收益。

### 决策 2 — Delegate 独占写权限

Parser 只「翻译」URL → State，**不持有可变栈**。  
否则会出现 Parser 与 Delegate 双写状态的竞态——这是自研路由最常见的坑之一。

### 决策 3 — 门面与内核分离

业务 App 只接触 `FastRouterConfig` + `FastNavigator`，不直接操作 Delegate。  
BarrelLock 的 `AppRouter` 单例就是这一决策的落地：

```dart
// 目标形态 — apps/BarrelLock/.../router/router.dart
class AppRouter {
  static RouterConfig<Object> get routerConfig => ...;

  static void push(String location) => FastNavigator.push(location);
  static void go(String location) => FastNavigator.go(location);
}
```

业务层甚至不知道 `NavigationState` 的存在——这正是 Facade 层的价值。

---

## 八、当时写下的「非目标」

刻意不做的事，避免 scope creep：

- ❌ 不做 Navigator 1.0 兼容层
- ❌ 不引入 code generation（与 auto_route 路线区分）
- ❌ M1 不做 Shell / Tab 嵌套栈
- ❌ 不以 `pageBuilder` 为主 API（M2 结论，见 06 篇）
- ❌ Modal / Dialog 不走 Router 栈（与 2.0 Page 栈正交）
