# Navigator 2.0 四件套核心复习

> **阶段定位**：实现 fast_navigator 之前的「针对性补课」——不是 Flutter 入门，而是把四件套的**职责边界**刻进肌肉记忆。  
> **原则**：只复习与自研路由直接相关的核心，略过 Restoration、Router.neglect 等边缘 API。

---

## 一、1.0 vs 2.0 — 为什么要换范式

| 维度 | Navigator 1.0 | Navigator 2.0 |
| --- | --- | --- |
| 栈管控 | `push/pop` 命令式改栈，开发者不可见 | `List<Page>` 声明式驱动，栈完全透明 |
| URL / Web | 需额外插件，割裂 | `RouteInformation` 一等公民 |
| 状态恢复 | 难 | Parser 可从 URL 重建栈 |
| 页面单元 | `Route`（命令式） | `Page`（声明式，可 diff） |
| 系统返回 | Navigator 内部处理 | `BackButtonDispatcher` + `popRoute()` 显式协议 |

**心路历程**：1.0 的「方便」是透支了可观测性。生产级 App 一旦涉及 Web 深链、登录拦截、混合栈，黑盒栈会成为 debug 噩梦。2.0 把栈变成**可 diff 的数据结构**，这才是 fast_navigator 的地基。

---

## 二、官方数据流（必须能默写）

```
浏览器 / 系统 URL 变化
        ↓
RouteInformationParser.parseRouteInformation()
        ↓
NavigationState（不可变）
        ↓
RouterDelegate.build()  →  Navigator(pages: [...])
        ↑
RouterDelegate.currentConfiguration
        ↓
RouteInformationParser.restoreRouteInformation()  →  回写 URL
```

**与 go_router 的关系**：go_router 不是替代 2.0，而是在此之上加了 RouteMatcher、Redirect、Shell。fast_navigator 对标这一层，底层仍走原生四件套。

## 三、四件套逐一拆解

### 3.1 Page — 最小渲染单元

本质：Navigator 维护的底层路由对象，包装了一层给 `Navigator` 使用。

核心字段（复习用，非完整 API）：

```dart
abstract class Page<T> extends RouteSettings {
  const Page({
    this.key,
    super.name,
    super.arguments,
    this.canPop = true,
    this.onPopInvoked = _defaultPopInvokedHandler,
  });

  /// 页面唯一标识 — canUpdate 据此判断是否同一页面
  final LocalKey? key;

  /// runtimeType 与 key 相同 → 原地更新，无需重建 Route
  bool canUpdate(Page<dynamic> other) {
    return other.runtimeType == runtimeType && other.key == key;
  }

  /// 创建对应 Route；Route.settings 须指向当前 Page
  @factory
  Route<T> createRoute(BuildContext context);
}
```

**fast_navigator 映射**：

- M1：`FastPage` 统一 `createRoute()` → `MaterialPageRoute`
- M2：`PageFactory` 按策略产出 `MaterialPage` / `CupertinoPage` / `CustomTransitionPage`（见 06 篇）

**关键认知**：动画不在 `Navigator.push` 里决定，而在 **`Page.createRoute()`**。改造动画 = 改造 Page 类型。

### 3.2 RouterDelegate — 唯一栈管理者

```dart
abstract class RouterDelegate<T> extends Listenable {
  /// 当前路由状态 — Router 监听此字段变化刷新页面
  T? get currentConfiguration;

  /// 根据路由状态构建 Navigator + List<Page>
  Widget build(BuildContext context);

  /// 系统返回时调用，处理页面出栈
  Future<bool> popRoute();
}
```

**fast_navigator 映射**：`FastRouterDelegate`

```dart
// 当时的设计伪代码 — packages/fast_navigator/lib/src/router/fast_router_delegate.dart
class FastRouterDelegate extends RouterDelegate<NavigationState>
    with ChangeNotifier, PopNavigatorRouterDelegateMixin<NavigationState> {

  NavigationState _state;  // 唯一可变源，通过不可变替换更新

  @override
  NavigationState? get currentConfiguration => _state;

  void updateState(NavigationState newState) {
    if (_state != newState) {
      _state = newState;
      notifyListeners();  // 触发 build
    }
  }

  @override
  Future<bool> popRoute() async {
    if (_state.matches.length > 1) {
      updateState(_state.pop());
      return true;   // 已消费返回事件
    }
    return false;    // 交给系统（如退出 App）
  }

  @override
  Widget build(BuildContext context) {
    final pages = _state.matches
        .map((m) => pageFactory.build(context: context, match: m))
        .toList();

    return Navigator(
      key: navigatorKey,
      pages: pages,
      onDidRemovePage: _onDidRemovePage,  // 手势返回同步 State
    );
  }
}
```

**三个必须混入的能力**：

1. `ChangeNotifier` — 状态变更通知 Router 重建
2. `PopNavigatorRouterDelegateMixin` — 绑定 `navigatorKey`，系统返回能找到 Navigator
3. `onDidRemovePage` — 手势 pop 时同步 `_state`（否则 State 与 UI 栈不一致）

### 3.3 RouteInformationParser — 状态 ⇄ URL 翻译器

```dart
abstract class RouteInformationParser<T> {
  /// 系统 URL / 深链接 → 业务路由状态 T
  Future<T> parseRouteInformation(RouteInformation routeInformation);

  /// 业务路由状态 T → 系统 RouteInformation
  RouteInformation? restoreRouteInformation(T configuration);
}
```

**fast_navigator 映射**：`FastRouteInformationParser`

```dart
// 设计伪代码
class FastRouteInformationParser extends RouteInformationParser<NavigationState> {
  final RouteRegistry registry;

  @override
  Future<NavigationState> parseRouteInformation(RouteInformation info) async {
    final uri = info.uri.path.isEmpty ? Uri(path: '/') : info.uri;
    final match = registry.matchLocation(uri);
    // Deep Link / Web 刷新：替换为单页栈
    return NavigationState(matches: [match]);
  }

  @override
  RouteInformation? restoreRouteInformation(NavigationState configuration) {
    if (configuration.isEmpty) return null;
    return RouteInformation(uri: configuration.location);  // 栈顶 URI
  }
}
```

**设计决策 — Location 模式（默认）**：

| 策略 | 行为 | 适用 |
| --- | --- | --- |
| Location 模式 | URL = 栈顶 location | 移动端主路径 + Web 友好 |
| Full Stack 模式 | URL 编码完整栈 | 强 Web 多 Tab 历史（未实现） |

Web 刷新时内存栈丢失 → 默认恢复为单页 `[match]`。中间 push 页若需恢复，须写入 URL 或接受产品约定。

### 3.4 RouterConfig + BackButtonDispatcher — 组装与系统返回

```dart
// MaterialApp.router 接入形态
MaterialApp.router(
  routerConfig: RouterConfig<Object>(
    routerDelegate: delegate,
    routeInformationParser: parser,
    routeInformationProvider: PlatformRouteInformationProvider(
      initialRouteInformation: RouteInformation(uri: Uri(path: '/')),
    ),
    backButtonDispatcher: RootBackButtonDispatcher(),
  ),
);
```

**fast_navigator 映射**：`FastRouter.routerConfig` 一键产出上述配置。

**BackButtonDispatcher 职责**：把 Android 物理返回键、Web 浏览器后退转发给 `delegate.popRoute()`。没有它，系统返回与 `FastNavigator.pop()` 行为不一致。

## 四、四件套协作时序（Push 一次页面）

```
业务调用 FastNavigator.push('/detail')
        ↓
registry.matchLocation('/detail') → RouteMatch
        ↓
delegate.state.push(match) → 新 NavigationState
        ↓
delegate.updateState(newState) → notifyListeners()
        ↓
Router 触发 delegate.build()
        ↓
PageFactory.build() → 新 Page 加入 List
        ↓
Navigator diff → 新 Page.createRoute() → 转场动画
        ↓
currentConfiguration 变化 → restoreRouteInformation → URL 回写（Web）
```

**注意**：整个链路**从不调用** `Navigator.push`。这是 2.0 与 1.0 的本质分界线。

## 五、自研时最容易踩的五个坑

| # | 坑 | 后果 | fast_navigator 对策 |
| --- | --- | --- | --- |
| 1 | 直接 mutate `matches` 列表 | Diff 不触发，页面不刷新 | `List.unmodifiable` + 返回新 `NavigationState` |
| 2 | Key 含 extra | extra 变一次 dispose 一次 | extra 不进 `RouteMatch.key` |
| 3 | Parser 与 Delegate 双写状态 | 竞态、栈错乱 | Parser 只翻译，Delegate 独占写 |
| 4 | 忽略 `onDidRemovePage` | 手势返回后 State 栈多一层 | 通过 `page.key` 同步移除 match |
| 5 | Modal 与 Router 栈混用 | 返回行为不可预测 | Dialog 仍用 `showDialog`，不走 Page 栈 |
