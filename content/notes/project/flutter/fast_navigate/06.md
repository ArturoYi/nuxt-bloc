# 路由动画设计与实现

> **阶段定位**：M2 核心交付——`builder + transition` 分离，框架管 Page 壳，业务只管 Widget。  
> **代码位置**：`packages/fast_navigator/lib/src/domain/transition/` + `lib/src/page/`  
> **验证标准**：`test/page_factory_test.dart` 五场景全绿

---

## 一、M1 的问题 — 为什么要改

M1 的 `FastPage.createRoute()` 写死了 `MaterialPageRoute`：

```dart
// M1 — 所有路由共用 Material 默认动画，无法按平台/路由配置
Route createRoute(BuildContext context) {
  return MaterialPageRoute(
    settings: this,
    builder: (ctx) => match.route.builder(ctx, match),
  );
}
```

| 问题 | 影响 |
| --- | --- |
| CupertinoApp 下仍走 Material 路由 | 动画与平台不一致 |
| 无法单路由禁用动画 | Splash 等场景需要零动画 |
| 无法 fade / slide 自定义 | 产品需求无法满足 |
| Page 类型写死 | Key / name / arguments 虽在 Page 层，扩展困难 |

---

## 二、心路历程：对齐 go_router，但不照搬 pageBuilder

对比 go_router 后发现：

- **默认路径**：`builder` → 平台适配 Page（MaterialPage / CupertinoPage / NoTransitionPage）
- **自定义路径**：`pageBuilder` → 完全接管 Page

| 方案 | 结论 |
| --- | --- |
| 以 `pageBuilder` 为主 | ❌ 业务样板多、Key 易漏 |
| **以 `builder + transition` 为主** | ✅ 内容 vs 过渡分离，框架管 Page 壳 |
| `pageBuilder` 作为逃生口 | ⏳ 留 M3，覆盖 barrier / restorationId 等 5% 场景 |

**核心原则**：业务只写 `builder`（Widget），框架通过 `PageFactory` 决定 Page 类型和动画。

### Navigator 2.0 动画的本质

动画不在 `Route` 命令式创建里决定，而在 **`Page.createRoute()`**：

```
List<Page>  diff  →  每个 Page.createRoute()  →  底层 PageRoute + transition
```

因此改造点不是改 `Navigator.push`，而是：**谁创建 Page、Page 用什么类型**。

---

## 三、设计目标

| 能力 | 说明 | M2 状态 |
| --- | --- | --- |
| 平台默认 | MaterialApp / CupertinoApp / WidgetsApp 自动选 Page | ✅ |
| 全局覆盖 | 应用级统一动画策略 | ✅ `defaultTransition` |
| 路由级覆盖 | 单条路由自定义 / 无动画 | ✅ `FastRoute.transition` |
| 完全接管 | `pageBuilder` 返回任意 Page | ⏳ M3 |
| 可测试 | 不依赖真实 Widget 树 | ✅ `appTypeOverride` |

---

## 四、解析优先级（三级链）

```
route.transition ?? config.defaultTransition ?? PlatformAdaptiveTransition
        ↓
PageFactory.switch(transition) → MaterialPage / CupertinoPage / ...
        ↓
child = route.builder(context, match)   // 业务 Widget 始终由 builder 产出
```

M2 不做 `pageBuilder`；M3 会在最前面加：`pageBuilder ?? 上述链`。

### 整体数据流

```
FastRoute(builder + transition?)
        ↓
FastRouterDelegate.build()
        ↓
PageFactory.build(context, match)
        ↓
MaterialPage / CupertinoPage / NoTransitionPage / CustomTransitionPage
        ↓
Navigator(pages: [...])
```

## 五、分步实现记录

### Step 1 — 定义过渡策略模型（Domain 层）

**想什么**

- 策略是「领域概念」，不应 import `material.dart` / `cupertino.dart`
- 用 `sealed class` 穷举，让 `PageFactory` 的 `switch` 编译期安全
- 自定义动画用 `CustomTransition + transitionsBuilder`，不让用户继承 sealed 类

**伪代码**

```dart
// domain/transition/page_transition.dart
// 仅依赖 flutter/widgets.dart

sealed class PageTransition {
  const PageTransition();
}

final class PlatformAdaptiveTransition extends PageTransition {
  const PlatformAdaptiveTransition();  // 默认
}

final class MaterialTransition extends PageTransition {}
final class CupertinoTransition extends PageTransition {}
final class NoTransition extends PageTransition {}

final class CustomTransition extends PageTransition {
  const CustomTransition({
    required this.transitionsBuilder,
    this.transitionDuration = const Duration(milliseconds: 300),
    this.fullscreenDialog = false,
    this.opaque = true,
  });

  final RouteTransitionsBuilder transitionsBuilder;
  final Duration transitionDuration;
  final Duration reverseTransitionDuration;
  final bool maintainState;
  final bool fullscreenDialog;
  final bool opaque;
}
```

---

### Step 2 — Page 基础设施（Adapter + 自定义 Page）

**想什么**

- Material / Cupertino 用 Flutter 原生 `MaterialPage` / `CupertinoPage`，不重复造轮子
- 自定义 / 无动画需要自己写 `Page` + `PageRoute`（参考 go_router `CustomTransitionPage`）
- **所有 Page 必须统一 Key 策略**，否则 Navigator diff 和 pop 栈同步会乱

**文件职责**

| 文件 | 职责 |
| --- | --- |
| `app_type_detector.dart` | 探测 MaterialApp / CupertinoApp / WidgetsApp，结果缓存 |
| `material_page_adapter.dart` | RouteMatch → MaterialPage |
| `cupertino_page_adapter.dart` | RouteMatch → CupertinoPage |
| `custom_transition_page.dart` | CustomTransition → PageRoute + transitionsBuilder |
| `no_transition_page.dart` | Duration.zero，transitionsBuilder 直接返回 child |

**伪代码**

```dart
// app_type_detector.dart
enum AppType { material, cupertino, widgets }

AppType detect(BuildContext ctx, {AppType? override}) {
  if (override != null) return override;
  if (ctx.findAncestorWidgetOfExactType<MaterialApp>() != null) return AppType.material;
  if (ctx.findAncestorWidgetOfExactType<CupertinoApp>() != null) return AppType.cupertino;
  return AppType.widgets;
}

// material_page_adapter.dart
MaterialPage build({required RouteMatch match, required Widget child}) {
  return MaterialPage(
    key: ValueKey(match.key),       // 框架统一 Key
    name: match.route.name,
    arguments: match.parameters,
    child: child,
  );
}

// custom_transition_page.dart
class CustomTransitionPage extends Page {
  Route createRoute(ctx) => _CustomTransitionPageRoute(this);
  // _CustomTransitionPageRoute extends PageRoute
  // buildTransitions → transitionsBuilder(context, animation, secondaryAnimation, child)
}

// no_transition_page.dart
class NoTransitionPage extends CustomTransitionPage {
  NoTransitionPage(...) : super(
    transitionsBuilder: (_, __, ___, child) => child,
    transitionDuration: Duration.zero,
  );
}
```

**平台默认动画效果（Flutter 框架行为，非我们手写）**

| App 类型 | Page | 动画 |
| --- | --- | --- |
| MaterialApp | MaterialPage → MaterialPageRoute | Android Material；iOS/macOS 内部 Cupertino 侧滑 |
| CupertinoApp | CupertinoPage → CupertinoPageRoute | 始终 iOS 水平侧滑 |
| WidgetsApp | NoTransitionPage | 无动画 |

### Step 3 — PageFactory 统一解析链

**想什么**

- 所有「match → Page」逻辑集中一处，Delegate 不再关心 Page 类型
- 先调 `builder` 拿 child，再包 Page —— **内容与过渡解耦**
- `switch (transition)` 配合 sealed，新增策略时编译器强制补全分支

**伪代码**

```dart
class PageFactory {
  final PageTransition defaultTransition;
  final AppType? appTypeOverride;

  Page build({required BuildContext context, required RouteMatch match}) {
    // 1. 解析策略：路由级 > 全局级
    final transition = match.route.transition ?? defaultTransition;

    // 2. 业务内容
    final child = match.route.builder(context, match);

    // 3. 策略 → Page
    return switch (transition) {
      PlatformAdaptiveTransition() =>
        _buildPlatformAdaptive(context, match, child),
      MaterialTransition() =>
        MaterialPageAdapter.build(match: match, child: child),
      CupertinoTransition() =>
        CupertinoPageAdapter.build(match: match, child: child),
      NoTransition() =>
        NoTransitionPage(match: match, key: ValueKey(match.key), child: child),
      CustomTransition(:final transitionsBuilder, ...) =>
        CustomTransitionPage(
          match: match,
          key: ValueKey(match.key),
          child: child,
          transitionsBuilder: transitionsBuilder,
        ),
    };
  }

  Page _buildPlatformAdaptive(ctx, match, child) {
    return switch (detect(ctx, override: appTypeOverride)) {
      AppType.material  => MaterialPageAdapter.build(...),
      AppType.cupertino => CupertinoPageAdapter.build(...),
      AppType.widgets   => NoTransitionPage(...),
    };
  }
}
```

### Step 4 — FastRoute 增加 transition 字段

```dart
final class FastRoute extends FastBaseRoute {
  final RoutePageBuilder builder;           // 必填：页面内容
  final PageTransition? transition;         // 可选：路由级过渡

  const FastRoute({
    required super.name,
    required super.path,
    required this.builder,
    this.transition,
    super.middlewares = const [],
  });
}
```

业务 API 保持简洁：普通路由只写 `builder`，零改动兼容 M1。

### Step 5 — 接入 Delegate / Router / Config

**关键变更**

1. `FastRouterDelegate` 原来 `FastPage(match: match)` → 改为 `_pageFactory.build()`
2. `onDidRemovePage` 原来依赖 `page is FastPage` → 改为读 `page.key`（ValueKey）
   - Page 可能是 MaterialPage、CustomTransitionPage 等，无法统一 `is` 判断
3. 配置从外到内传递：`FastRouterConfig` → `FastRouter` → `FastRouterDelegate` → `PageFactory`

**伪代码**

```dart
// fast_router_config.dart — 对外暴露
FastRouterConfig(
  routes: [...],
  defaultTransition: const PlatformAdaptiveTransition(),
  appType: null,  // 单测可注入 AppType.material
)

// fast_router.dart — 组装
delegate = FastRouterDelegate(
  pageFactory: PageFactory(
    defaultTransition: defaultTransition,
    appTypeOverride: appType,
  ),
);

// fast_router_delegate.dart — build
final pages = _state.matches
    .map((m) => _pageFactory.build(context: context, match: m))
    .toList();

Navigator(pages: pages, onDidRemovePage: _onDidRemovePage);

// onDidRemovePage — 通过 Key 找 match
String? _pageKeyOf(Page page) {
  if (page.key case ValueKey<String>(:final value)) return value;
  return null;
}
```

### Step 6 — 移除旧 FastPage + 测试

**想什么**

- `FastPage` 职责已被 `PageFactory` 完全取代，保留会造成双路径
- 单测聚焦 `PageFactory` 策略解析，不测 Flutter 框架内部动画像素
- 用 `appTypeOverride` 跳过 Widget 树 walk，测试稳定

## 六、业务侧用法

```dart
// 默认 — 零配置，跟随 App 类型
FastRoute(
  name: 'home',
  path: '/',
  builder: (_, m) => HomePage(),
)

// 单路由无动画（Splash）
FastRoute(
  name: 'splash',
  path: '/splash',
  transition: const NoTransition(),
  builder: (_, m) => SplashPage(),
)

// 单路由自定义 fade
FastRoute(
  name: 'modal',
  path: '/modal',
  transition: CustomTransition(
    fullscreenDialog: true,
    transitionsBuilder: (ctx, anim, _, child) =>
        FadeTransition(opacity: anim, child: child),
  ),
  builder: (_, m) => ModalPage(),
)

// 全局强制 Cupertino 侧滑
FastRouterConfig(
  defaultTransition: const CupertinoTransition(),
  routes: [...],
)
```

## 八、与 go_router 对照

| 维度 | go_router | fast_navigator M2 |
| --- | --- | --- |
| 默认 | `builder` + platform adapter | `builder` + `PageFactory` |
| 路由自定义动画 | `pageBuilder` + CustomTransitionPage | `transition: CustomTransition` |
| 全局默认 | 无内置，需 helper | `defaultTransition` 一等公民 |
| 策略建模 | 隐式 | 显式 `sealed PageTransition` |
| 完全接管 Page | `pageBuilder` | M3 待办 |
